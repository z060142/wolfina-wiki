# 🦊 LLM 專用 Wiki 系統（含多 Agent 協作）— 需求規格

---

# 1. 系統定位

本系統為一個：

> **由多個 LLM Agent 協作維護的內部知識系統**

其目標為：

* 建立可被 LLM 長期維護的知識庫
* 透過多 Agent 機制降低錯誤與知識污染
* 提供可追溯、可驗證、可控的知識演進
* 透過插件層與其他系統協作，同時保持主系統獨立運作

---

# 2. 核心設計原則

## 2.1 不信任單一 Agent

任何單一 Agent 的輸出都不可直接寫入正式知識。

## 2.2 所有修改必須經過多方驗證

至少需經過：

* 生成（Proposer）
* 審核（Reviewer）
* 套用（Executor）

## 2.3 知識必須可追溯

所有變更需可追蹤來源：

* 哪個 Agent
* 哪個任務
* 哪個理由

## 2.4 寧可延遲，不可污染

資料一致性優先於即時更新。

## 2.5 主系統必須獨立

即使所有插件停用、失敗或被移除，主系統仍必須維持基本功能：

* Page 管理
* 查詢
* 多 Agent 提案 / 審核 / 套用流程
* Version 紀錄
* Relation 管理

## 2.6 擴充能力不得反向綁死核心

任何外部協作能力必須透過插件層接入，不得使核心資料模型、核心流程或主系統可用性依賴單一外部系統。

---

# 3. 系統角色（Agent Roles）

---

## 3.1 Proposer Agent（提案者）

負責：

* 建立新頁面
* 修改既有頁面
* 提供修改理由（rationale）
* 提供來源（若有）

限制：

* 不得直接修改正式內容

---

## 3.2 Reviewer Agent（審核者）

負責：

* 驗證提案內容
* 檢查邏輯一致性
* 檢查是否與既有知識衝突
* 判斷是否接受修改

輸出：

* approve / reject
* optional feedback

---

## 3.3 Executor Agent（執行者）

負責：

* 套用已通過的提案
* 建立新版本
* 更新關係與索引

限制：

* 僅能執行已核准提案
* 不可自行修改內容

---

## 3.4 Curator Agent（維護者，可選）

負責：

* 清理過時內容
* 合併重複頁面
* 修正關係錯誤
* 標記低品質內容

---

## 3.5 Auditor Agent（稽核者，可選）

負責：

* 定期檢查知識品質
* 偵測矛盾內容
* 偵測低信心資料
* 產生修復任務

---

## 3.6 Plugin Agent / Integration Agent（插件協作代理，可選）

負責：

* 透過插件層與外部系統互動
* 擷取、同步、轉換或推送資料
* 觸發主系統內的標準流程（如建立提案、附加來源、建立關聯）

限制：

* 不得繞過主系統的 Proposal / Review / Apply 流程
* 不得直接修改核心資料
* 不得要求主系統依賴外部系統才能運作

---

# 4. 功能需求

---

## 4.1 Page 管理

系統必須支援：

* 建立 Page
* 讀取 Page
* 封存 Page
* 不允許直接覆寫

---

## 4.2 Page 結構

每個 Page 必須包含：

* title
* slug（唯一）
* content
* summary
* updated_at

建議欄位：

* canonical_facts
* source_refs
* confidence

---

## 4.3 關係管理（Relation）

系統必須支援：

### 必要關係

* parent
* child
* related_to
* references

### 功能

* 查詢關聯頁
* 查詢子頁
* 查詢反向關聯

---

## 4.4 查詢能力

系統必須支援：

### 基本查詢

* id / slug
* title 搜尋

### 結構查詢

* 子頁
* 關聯頁

### 條件查詢

* 最近更新
* 指定狀態

### 語意查詢（建議）

* embedding search

---

# 5. 修改流程（核心）

---

## 5.1 提案階段（Propose）

Proposer Agent 提交：

* target_page
* proposed_content
* proposed_summary
* rationale
* source_refs（可選）

產生：
→ EditProposal（狀態：pending）

---

## 5.2 審核階段（Review）

Reviewer Agent 必須：

* 檢查內容正確性
* 檢查與既有內容衝突
* 檢查是否合理

輸出：

* approve
* reject
* feedback（可選）

---

## 5.3 套用階段（Apply）

Executor Agent：

* 僅處理 approved 提案
* 更新 Page
* 建立新版本
* 更新 relation / index

---

## 5.4 強制規則

* 未經審核不得寫入
* 不允許直接修改 Page
* 所有修改必須有提案紀錄

---

# 6. 版本管理

系統必須：

* 保存所有版本
* 記錄：

  * 編輯來源 Agent
  * 編輯時間
  * 編輯理由

---

# 7. 多 Agent 協作規則

---

## 7.1 分離職責（Separation of Duties）

* proposer ≠ reviewer ≠ executor
* 同一 Agent 不可同時擔任三角色

---

## 7.2 最低審核要求

每個提案必須：

* 至少 1 reviewer 通過

（可擴充為多 reviewer 共識）

---

## 7.3 衝突處理

若 reviewer 判定：

* 與現有內容衝突

→ 必須：

* 標記衝突
* 或建立對立頁面
* 或拒絕提案

---

## 7.4 多提案競爭

若多個提案修改同一頁：

系統必須：

* 支援版本分支（logical）
* 或序列化處理

---

## 7.5 信任機制（可選）

系統可引入：

* agent reputation
* trust score

用於：

* 決定是否需要更多 reviewer
* 決定自動通過門檻

---

# 8. 插件層（Plugin Layer）

---

## 8.1 插件層定位

系統必須提供一個插件層，用於擴充與外部系統的協作能力。
插件層的目的在於：

* 連接外部資料來源
* 連接外部任務系統
* 連接外部檢索、通知、同步或分析能力
* 為主系統增加協作能力，但不改變主系統核心職責

---

## 8.2 插件層基本原則

插件層必須符合以下原則：

* **非核心依賴**：主系統不得依賴任何單一插件才能正常運作
* **隔離失敗**：插件失敗不得導致主系統核心流程失效
* **受控接入**：插件不得直接繞過核心寫入治理流程
* **可停用**：任一插件都必須可被獨立啟用、停用、移除
* **最小侵入**：插件不得任意改寫核心資料結構

---

## 8.3 插件類型

系統應允許以下類型插件：

### 外部資料來源插件

例如：

* 文件系統
* API 來源
* 外部知識庫
* issue tracker
* chat / message 系統

用途：

* 匯入資料
* 監聽更新
* 附加來源引用

### 外部輸出插件

例如：

* 通知系統
* 報表系統
* 外部索引系統
* 搜尋或分析平台

用途：

* 推送變更
* 傳送事件
* 生成外部報表或摘要

### 協作插件

例如：

* 任務分派
* 工作流整合
* 外部審核或標註系統

用途：

* 建立外部任務
* 接收外部審核訊號
* 對接其他代理系統

---

## 8.4 插件能力邊界

插件可以：

* 讀取主系統公開資料
* 提交標準化請求
* 建立或補充 Proposal
* 附加 metadata、source refs、外部事件記錄
* 觸發主系統內部的合法流程

插件不可以：

* 直接改寫 Page 正式內容
* 直接改寫 Version 歷史
* 跳過 Reviewer 審核
* 修改核心規則
* 破壞主系統資料一致性

---

## 8.5 插件失效處理

系統必須保證：

* 單一插件故障不影響主系統核心查詢
* 單一插件故障不影響 Proposal / Review / Apply 流程
* 插件逾時、失敗、資料異常時，系統可記錄錯誤並隔離影響
* 插件恢復後可重新接入，不要求主系統重建

---

## 8.6 插件生命週期管理

系統應支援：

* 註冊插件
* 啟用插件
* 停用插件
* 移除插件
* 查詢插件狀態
* 記錄插件版本與能力範圍

---

## 8.7 插件事件模型

插件層應支援事件導向協作。
建議支援的事件包括：

* page_created
* page_updated
* proposal_created
* proposal_reviewed
* proposal_applied
* page_archived
* relation_added

插件可訂閱事件，但事件投遞失敗不得阻塞主系統核心流程，除非明確標記為非核心同步任務。

---

## 8.8 插件安全與隔離

系統應具備：

* 插件權限邊界
* 插件操作日誌
* 插件來源識別
* 插件行為追蹤
* 插件輸入驗證

任何由插件發起的寫入動作，都必須保留：

* 插件 ID
* 呼叫來源
* 觸發時間
* 目標資源
* 結果狀態

---

## 8.9 插件與多 Agent 協作整合

插件可作為多 Agent 體系中的輔助層，但不得取代核心治理角色。

例如：

* 外部資料插件可觸發 Proposer Agent 建立提案
* 外部分析插件可作為 Reviewer Agent 的參考來源
* 外部任務插件可接收 Curator / Auditor 產生的修復任務

但最終：

* 正式寫入仍需經主系統審核
* 主系統的版本與追溯紀錄仍為唯一準據

---

# 9. 驗證與品質控制

---

# 11. 最小可行版本（MVP）

必須包含：

* Page 系統
* Relation（基本）
* 搜尋（title + keyword）
* Proposal 機制
* Reviewer 審核
* Executor 套用
* Version 紀錄
* 基本插件層框架
* 插件啟用 / 停用能力
* 插件事件通知能力（至少非阻塞）

---

# 12. 成功標準

系統必須達成：

* LLM 可穩定查詢知識
* LLM 可安全更新知識
* 知識不因單一 Agent 而污染
* 所有變更可追溯
* 關聯查詢可用
* 插件可安全擴充外部協作能力
* 插件失效不影響主系統核心運作

---

# 13. 已知風險

* Agent 間共識失敗
* reviewer 判斷錯誤
* 知識逐步偏移
* 過度審核導致更新延遲
* 插件輸入品質不穩定
* 外部系統異常造成雜訊或錯誤事件輸入

---

# 🦊 結案備註

本系統本質為：

> **一個由多個 LLM Agent 互相制衡與協作維護，且可透過插件層安全擴充外部協作能力的知識系統**

而非傳統 wiki。

其核心價值在於：

* 控制知識寫入
* 建立可信任的長期記憶
* 避免模型集體幻覺
* 在不破壞主系統穩定性的前提下，與其他系統協作
